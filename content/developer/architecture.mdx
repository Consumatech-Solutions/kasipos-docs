# Project Architecture

KasiPOS is built with a **local-first, offline-aware architecture** designed to operate reliably in low-connectivity or no-connectivity environments, while maintaining a high-performance retail experience. Every structural decision — from directory layout to layer separation — serves this operational reality.

## 1. Architectural Evolution: From Monorepo to Single Repo

KasiPOS was originally designed around a classic monorepo structure. To eliminate deployment overhead and inter-package versioning friction, those concepts have been **internalized and distributed** within a unified Next.js 15 application — one codebase, one deployment, one team.

### Concept Mapping

| Monorepo Concept | Current Implementation | Rationale |
|---|---|---|
| `apps/` (multiple apps) | `src/app/` (Next.js App Router) | Focused on in-store POS operations; admin and mobile can be separate repos if ever needed |
| `packages/shared-ui` | `src/components/ui/` + Radix UI | Component library lives inline — no external package compilation overhead |
| `packages/api-clients` | `src/lib/api/` | API layer is domain-specific to POS operations |
| `packages/utilities` | `src/lib/utils/` + `src/lib/services/` | Utilities co-located with their usage contexts |
| `packages/types` | `src/types/` | Shared TypeScript definitions for the entire application |
| `docs/` | `docs/` (root, preserved) | Documentation in `docs/en/` and `docs/fr/` |
| `public/` | `public/` (root, preserved) | Critical PWA assets: `manifest.json`, `sw.js`, icons |

**Why this evolution?**

- **Operational focus** — The project specifically targets in-store POS operations, not a multi-application platform.
- **Deployment simplicity** — A single Next.js deployment reduces complexity and operational overhead.
- **Team efficiency** — No inter-package dependency management or versioning to maintain.
- **Offline-first priority** — The local-first architecture demands tight coupling between the data layer and UI components.

#### Decoding the Offline Engine

      <div><strong className="font-mono">entity-cache.ts</strong> — High-level caching abstraction on top of IndexedDB. Provides read/write operations that components and hooks consume.       <div><strong className="font-mono">api/</strong> — HTTP network clients. Called only when connectivity is confirmed. Fresh data from the network feeds back into the local cache via <code>entity-cache.ts</code>, keeping both layers in sync.</div>
    </div>

      <div><strong className="font-mono">mutation-registry.ts</strong> — Central registry of mutation handlers. Maps each mutation type (e.g., <code>CREATE_PRODUCT</code>, <code>UPDATE_CUSTOMER</code>) to its processing function for re-synchronization.</div>
      <div><strong className="font-mono">cloud-data-pull.ts</strong> — Orchestrates background cloud synchronization. Triggers mutation queue flushing and fresh data fetching once connectivity is restored — automatically, without user intervention.</div>

## 5. Database Schema

The core database is managed by Dexie.js wrapping IndexedDB. Below is the foundational schema — each domain extends this base pattern with its own table definitions.

```typescript
import Dexie, { Table } from 'dexie';

export interface Sale {
  id?: number;
  total: number;
  timestamp: number;
  items: any[];
  synced: boolean;
}

export class KasiDatabase extends Dexie {
  sales!: Table<Sale>;

  constructor() {
    super('KasiPOS_DB');
    this.version(1).stores({
      // ++id = auto-increment primary key
      // timestamp, synced = indexed fields for queries
      sales: '++id, timestamp, synced'
    });
  }
}

export const db = new KasiDatabase();
```

The `synced` boolean field is central to the offline architecture — it flags which records have been pushed to the cloud and which are still pending. The background sync worker queries `synced: false` to build its upload batch.

### Layer responsibility

| Layer | Directory | Responsibility |
|---|---|---|
| UI | `components/` | Renders data to the screen. No direct API calls. No IndexedDB access. Consumes hooks for all state. |
| Business Logic | `hooks/` | Business logic with React lifecycle. Wraps `lib/` functions with React state, effects, and TanStack Query. |
| Services | `lib/services/` | Stateless service functions for complex operations (image sync, device management, receipt generation). |
| Data Access | `lib/api/` + `lib/*.ts` | HTTP clients, IndexedDB schema, entity cache, mutation queue. No React dependencies. |
